## Security & Compliance

**sec-approved-package-sources** — Install packages only from approved, reputable registries  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: Dependencies **must** be installed only from approved package sources: - the **canonical public registry** for each ecosystem (e.g. npmjs.org, NuGet.org, PyPI); **or** - an **org-approved internal registry or proxy** for first-party and vetted packages. Packages **must not** be pulled directly from an individual publisher's personal source — arbitrary `git`/URL installs, personal or unvetted third-party registries, or hand-shared archives.

**sec-local-vulnerability-scanning** — Scan for vulnerabilities locally before pushing  
`Handbook` · `pre-commit` · agent: `align` · ADVISORY  
Rule: Developers **should** scan code locally before committing and pushing, covering the same three surfaces the remote gate checks — static analysis (SAST), dependency vulnerabilities, and secrets — and resolve anything **above Low** severity before pushing to the remote. Recommended local toolset: - **Semgrep** (SAST) + **Trivy** (dependency / container CVEs) + **gitleaks** (secrets), wired through a pre-commit hook; **or** - the **CodeQL CLI** locally for SAST, when you want local findings to match the remote GHAS gate exactly (still pair it with a dependency and secrets scanner).

**sec-no-plaintext-secrets** — Never commit plaintext secrets to source control  
`Policy` · `ci-gate` · agent: `enforce` · BLOCK · refs: SOC2 CC6.1, ISO 27001 A.9.2.3, NIST SP 800-53 IA-5  
Rule: Credentials, API keys, tokens, private keys, and connection strings **must not** be committed to source control in plaintext. Secrets are supplied at runtime from an approved secret manager or the platform's encrypted configuration.

**sec-remote-vulnerability-scanning** — Block merges on unresolved vulnerabilities above Low  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Remote repositories **must** run vulnerability scanning as a required status check, and a pull request **must not** merge while any finding **above Low** severity (Medium, High, or Critical) is unresolved. **GitHub Advanced Security (GHAS)** is the approved scanner — code scanning (CodeQL), dependency scanning (Dependabot), and secret scanning. "Resolved" means the finding is fixed **or** formally dismissed with a recorded reason (risk-accepted), so genuinely unfixable transitive issues can be triaged rather than blocking indefinitely.

## Architecture & Tech Stack

**arch-adrs** — Record significant decisions as ADRs  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: Significant or hard-to-reverse architectural decisions **should** be captured as Architecture Decision Records (ADRs) checked into the codebase — the context, the options weighed, the decision, and its consequences.

**arch-api-first-design** — Design the API contract first  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: New services and integrations **should** start from an explicit API contract (the interface and its schema) agreed before the implementation is built, rather than letting the contract emerge from the code after the fact.

**arch-api-versioning** — Version APIs and preserve backward compatibility  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: Published APIs **should** carry an explicit version and evolve without breaking existing consumers: add rather than remove or repurpose fields, and introduce a new version when a breaking change is unavoidable, retiring the old one on a communicated timeline.

**arch-approved-app-stack** — Build on approved frontend and API stacks  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: New applications **should** be built on an approved stack: - **Frontend:** React, Angular, or Vue. - **APIs / services:** Node.js, Python, or .NET. This rule expresses *preference* among the approved options. The hard prohibition on introducing a framework or language outside the set lives in [`arch-no-unapproved-stack`](no-unapproved-stack.md).

**arch-approved-compute-platform** — Prefer AWS and serverless; containerize with Docker when needed  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: New workloads **should** run on AWS, with a **serverless-first** preference. When a serverless model doesn't fit, containerize with **Docker** as the container platform rather than provisioning bespoke long-lived hosts.

**arch-approved-datastores** — Prefer Postgres, file/document storage, or no database  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: When persistence is required, reach for an approved option in this order of preference: 1. **PostgreSQL** — the default for relational and general-purpose data. 2. **File / document storage** (e.g. XML or JSON documents) — for lightweight or configuration-shaped data that doesn't need a relational engine. 3. **No database** — prefer statelessness; don't introduce a datastore a service doesn't need.

**arch-diagrams-in-repo** — Keep architecture diagrams in the codebase  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: A service's **Capability**, **Logical**, and **Structural** architecture diagrams **should** live in its codebase (as diagram-as-code checked in alongside the source), so they version with the code and are reviewed in the same pull requests.

**arch-infrastructure-as-code** — Provision infrastructure as code with Terraform  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Cloud infrastructure **must** be defined as code with **Terraform** and applied through review, not provisioned by hand in a console. Infrastructure-as-code in another tool is not the standard, and manual, un-tracked infrastructure changes are not permitted as the way infrastructure is managed.

**arch-no-local-sql-databases** — No SQLite or local/embedded SQL databases  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: Services **must not** use SQLite or other local/embedded SQL engines, or developer-local SQL instances, as a datastore for application data. Relational persistence uses the approved managed database (PostgreSQL — see [`arch-approved-datastores`](approved-datastores.md)).

**arch-no-unapproved-stack** — No frontend frameworks or API languages outside the approved set  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: A new application **must not** introduce a frontend framework or API language outside the approved set: - **Frontend frameworks:** React, Angular, Vue. - **API / service languages:** Node.js, Python, .NET. Anything outside these sets is not permitted for a new project without an approved exception. The ranked *preference* among the approved options lives in [`arch-approved-app-stack`](approved-app-stack.md); this rule is the hard boundary around them.

**arch-service-boundaries** — Cross service boundaries through APIs, not shared databases  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: A service **should** own its data and expose it only through its API. Other services must not read or write its database directly or reach into its internal storage to share data.

## Quality & Testing

**qual-accessibility-checks** — UI must pass automated accessibility checks  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: User interfaces **must** pass automated accessibility checks (e.g. an axe-based scan) run in CI as a required check, with no unresolved violations at or above the agreed severity.

**qual-deterministic-tests** — Tests must be deterministic and isolated  
`Handbook` · `audit` · agent: `enforce` · ADVISORY  
Rule: Tests **should** be deterministic and isolated — the same input produces the same result, with no dependence on wall-clock time, network, ordering, or shared mutable state. A test that goes flaky is quarantined and fixed, not papered over by retrying until it goes green.

**qual-no-skipped-tests** — No skipped, disabled, or focused tests merged  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: Tests **must not** be merged in a skipped or disabled state without a tracked reason, and focused-test markers **must not** be merged at all: - No `.skip` / `xit` / `@Ignore` / `@Disabled` without a linked issue explaining why and when it returns. - No focused tests — `it.only`, `fdescribe`, `test.only` — ever, since they silently disable every other test in the file.

**qual-regression-test-for-bugfix** — Add a regression test with every bug fix  
`Handbook` · `human-review` · agent: `enforce` · ADVISORY  
Rule: When fixing a bug, you **should** add a test that fails before the fix and passes after it. Prefer writing the failing test first so the fix is demonstrably what closes it.

**qual-ui-automated-testing** — UI development must have automated tests  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: User-interface development **must** be covered by automated tests that run in CI as a required check. The approved UI testing tool is **Playwright** (specified under Integrations & Tooling). A UI change should ship with the automated tests that exercise it.

**qual-unit-test-coverage** — Meet the unit-test coverage bar on new code  
`Strategic` · `ci-gate` · agent: `enforce` · BLOCK  
Rule: New and changed code **must** reach at least **80% coverage**, and a pull request **must not** lower the repository's overall coverage. Coverage is measured on the diff (patch coverage), not by holding the whole legacy codebase to the bar. Prefer **branch coverage** where the tooling supports it, and exclude generated code and migrations.

## Delivery & CI/CD

**cicd-decompose-work** — Decompose work into small, independently branched pieces  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: A plan **should** be split into disparate, independently shippable pieces, with a separate branch (and pull request) for each piece, rather than developed as one large branch that lands everything at once.

**cicd-delete-merged-branches** — Delete branches after they merge  
`Handbook` · `ci-gate` · agent: `align` · WARN (CI, non-blocking)  
Rule: Topic branches **should** be short-lived and deleted once their pull request merges, rather than lingering. Prefer enabling the platform's automatic-delete-on-merge setting.

**cicd-deploy-via-pipeline** — Deploy through the pipeline, not by hand  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Deployments **must** go through the delivery pipeline, promoting a build through environments (dev → staging → production) rather than being applied by hand. Production deployments **must** pass an explicit approval gate.

**cicd-github-flow-branching** — Follow GitHub Flow for branching  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Teams **must** follow **GitHub Flow**: `main` is always deployable; work happens on short-lived branches cut from `main`; changes land back on `main` through a reviewed pull request. Long-lived parallel branches (a permanent `develop`, release trains) are not the standard model.

**cicd-post-deploy-validation** — Validate deployments after release  
`Handbook` · `audit` · agent: `aware` · ADVISORY  
Rule: A deployment **should** run automated post-deploy validation — smoke and health checks against the target environment — before it's considered complete, and a failing check **should** halt promotion or trigger a rollback rather than leaving a bad release live.

**cicd-required-checks-before-merge** — Gate every merge on required CI checks  
`Strategic` · `ci-gate` · agent: `aware` · BLOCK  
Rule: Changes to a protected branch **must** merge through a pull request whose required status checks (build, tests, and the guidance freshness gate) have passed. Direct pushes that bypass required checks are not permitted.

**cicd-required-peer-review** — Require an approving peer review before merge  
`Strategic` · `ci-gate` · agent: `aware` · BLOCK  
Rule: A pull request into a protected branch **must** have at least one approving review from someone other than the author before it can merge. Self-approval does not satisfy the requirement.

**cicd-safe-rollback** — Every release must have a safe rollback path  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: Every release **should** have a known, tested way to get back to a good state — a roll-back to the prior version or a roll-forward fix on a defined path — rather than discovering the recovery route during an incident. Changes that can't be trivially reverted (e.g. destructive migrations) should ship with their recovery plan.

**cicd-semver-release-tags** — Tag releases with a semantic version  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Every release **must** carry a **semantic version** tag of the form `MAJOR.MINOR.PATCH` (e.g. `v2.4.1`), incremented by the nature of the change: MAJOR for breaking changes, MINOR for backward-compatible additions, PATCH for backward-compatible fixes.

**cicd-small-changesets** — Prefer small commits and pull requests  
`Handbook` · `ci-gate` · agent: `align` · WARN (CI, non-blocking)  
Rule: Keep changes small: - **Commits** — prefer under **200 lines** changed; a commit should be one coherent step. - **Pull requests** — prefer under **1000 lines** changed; a PR should be one reviewable unit. Larger commits and PRs are discouraged and should be split where practical.

## Observability & Data

**obs-analytics-event-schema** — Follow an agreed schema for analytics events  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: Product analytics events **should** follow an agreed schema (a tracking plan): consistent event names, defined properties, and stable types — rather than ad-hoc events invented per feature.

**obs-correlation-ids** — Propagate correlation IDs across services  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: Services **should** generate a correlation (trace) id at the edge and propagate it through downstream calls, attaching it to logs and, where available, distributed traces — so a single request can be followed across service boundaries.

**obs-data-classification** — Classify data and handle it by its class  
`Strategic` · `audit` · agent: `aware` · REQUIRED (unenforceable — review/culture)  
Rule: Data **must** be classified (e.g. public / internal / confidential / regulated-PII) and handled according to its class — where it may be stored, who may access it, and how it must be protected. New data stores and flows must record the classification of what they hold.

**obs-health-readiness-endpoints** — Expose health and readiness endpoints  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: Services **should** expose **health** (is the process alive) and **readiness** (can it serve traffic) endpoints, so the platform and the delivery pipeline can tell whether an instance is healthy.

**obs-logs-to-approved-sink** — Emit logs and telemetry to an approved sink  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: Application logs, metrics, and traces **should** be emitted to an approved central sink using the standard structured format, rather than relying solely on local files or ad-hoc destinations for operational telemetry.

**obs-no-pii-in-logs** — Keep secrets and PII out of logs and telemetry  
`Strategic` · `audit` · agent: `enforce` · REQUIRED (unenforceable — review/culture)  
Rule: Logs, metrics, traces, and error payloads **must not** contain secrets or personal data (PII). Redact, mask, or omit sensitive fields before they're emitted rather than relying on the sink to hide them.

**obs-structured-logging** — Emit structured logs with consistent fields  
`Handbook` · `audit` · agent: `enforce` · ADVISORY  
Rule: Services **should** emit **structured** logs (e.g. JSON) with a consistent set of fields — level, timestamp, service name, and a correlation id — rather than free-form text lines. Avoid unstructured `print`-style logging in application code.

**obs-telemetry-retention** — Define retention windows for telemetry  
`Strategic` · `audit` · agent: `aware` · REQUIRED (unenforceable — review/culture)  
Rule: Logs, metrics, and traces **must** have a defined retention window — kept long enough to debug and audit, then expired — rather than being retained indefinitely or discarded ad hoc.

## Integrations & Tooling

**integ-ai-usage-guardrails** — Use only approved AI tools with company code  
`Strategic` · `human-review` · agent: `align` · REQUIRED (unenforceable — review/culture)  
Rule: Proprietary code, secrets, and confidential or personal data **must not** be pasted into or processed by AI tools outside the approved set — the approved AI assistant is **Claude** (see [`integ-approved-tools`](approved-tools.md)). Company work with AI runs through approved tooling, not arbitrary consumer AI services.

**integ-approved-tools** — Use the approved tool for each capability  
`Handbook` · `human-review` · agent: `align` · ADVISORY  
Rule: Reach for the approved tool for each capability before introducing an alternative: | Capability | Approved tool | |---|---| | Remote repository / VCS hosting | **GitHub** | | CI/CD platform | **GitHub Actions** | | AI assistant + model set | **Claude** (preferred) | | Diagramming | **Mermaid** | | Knowledge base | **Notion** | | UI testing | **Playwright** | Package registries are governed separately by [`sec-approved-package-sources`](../security-compliance/approved-package-sources.md). Adding a tool outside this list follows [`integ-new-tool-approval`](new-tool-approval.md); tools on the prohibited list are covered by [`integ-denylist-tools`](denylist-tools.md).

**integ-denylist-tools** — Prohibited tools must not be used  
`Policy` · `ci-gate` · agent: `enforce` · BLOCK · refs: SOC2 CC6.8, ORG-SEC-021  
Rule: Any tool, service, or dependency on the organization's **"DO NOT USE"** list **must not** be used, introduced, or integrated. The canonical list is the org's tooling denylist register (control `ORG-SEC-021`); the categories below describe what that register covers. | Prohibited (category) | Reason | |---|---| | Unlicensed / copyleft-incompatible tools | legal / licensing | | Tools with known unpatched CVEs or supply-chain compromise | security | | Services that store data outside approved regions | data-residency |

**integ-dependency-licenses** — Third-party dependencies must use allowed licenses  
`Strategic` · `ci-gate` · agent: `align` · BLOCK  
Rule: Third-party dependencies **must** carry an allowed open-source license. Dependencies under prohibited licenses (e.g. strong copyleft pulled into proprietary distribution) **must not** be introduced. The allowed/prohibited license sets are maintained centrally.

**integ-new-tool-approval** — Route new tools through the approval path  
`Handbook` · `human-review` · agent: `aware` · ADVISORY  
Rule: Adopting a tool not already on the approved list **should** go through the tool-approval path — a review of licensing, security, data handling, and overlap with existing tools — before it's introduced. The outcome updates the approved list or the "DO NOT USE" list so both stay current.

**integ-pin-third-party-actions** — Pin third-party CI actions and dependencies by digest  
`Handbook` · `ci-gate` · agent: `enforce` · WARN (CI, non-blocking)  
Rule: Third-party CI actions and build-time dependencies **should** be pinned to an immutable identifier (a commit SHA or content digest) rather than a mutable tag like `latest` or a floating major version.

**integ-tool-telemetry-off** — Disable tool telemetry and usage tracking  
`Strategic` · `human-review` · agent: `enforce` · REQUIRED (unenforceable — review/culture)  
Rule: Development and delivery tools **must** be configured with telemetry, usage tracking, and advertising/analytics features **turned off** where the tool allows it. The organization does not opt in to having its tool usage monitored by vendors.

## Developer Environment

**env-gitignore-local-settings** — Don't commit local environment settings  
`Handbook` · `pre-commit` · agent: `enforce` · ADVISORY  
Rule: Machine- and user-specific environment settings **should not** be committed — personal editor/IDE settings, local tool config, OS cruft (`.DS_Store`), and local `.env` files. Keep them out of source control with an appropriate `.gitignore` (and a shared org baseline where one exists).

**env-local-secrets-handling** — Handle local secrets safely  
`Handbook` · `pre-commit` · agent: `enforce` · ADVISORY  
Rule: Local development secrets **should** be supplied through a gitignored `.env` file or a local secret manager, never hardcoded into source and never committed. Provide a checked-in `.env.example` with placeholder keys so contributors know what to set without exposing real values.

**env-pinned-tool-versions** — Pin local runtime and tool versions  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: Repositories **should** pin the local runtime and tool versions they expect — via a version file such as `.nvmrc`, `.tool-versions`, or a `mise`/`asdf` config — so every contributor and CI run uses the same toolchain.

**env-pre-commit-hooks** — Provide standard pre-commit hooks  
`Handbook` · `pre-commit` · agent: `align` · ADVISORY  
Rule: Repositories **should** ship a standard pre-commit hook configuration that runs the formatter/linter and the local security scans before a commit is created, so problems are caught at the keyboard rather than at the pull-request gate.

**env-preferred-editors** — Prefer VS Code or the terminal as the editor  
`Handbook` · `none` · agent: `aware` · ADVISORY  
Rule: **VS Code** or a **terminal**-based editor are the preferred development environments. Teams should standardize shared editor configuration (extensions, settings) around these rather than each person maintaining a bespoke setup.

**env-reproducible-setup** — Make local setup reproducible  
`Handbook` · `audit` · agent: `align` · ADVISORY  
Rule: A repository **should** provide a reproducible way to stand up its local environment — a devcontainer, or a documented one-command bootstrap (e.g. `make setup`) — rather than a wiki page of manual steps that drifts out of date.

**env-shared-formatter-config** — Use the shared formatter and linter configuration  
`Handbook` · `pre-commit` · agent: `enforce` · ADVISORY  
Rule: Projects **should** adopt the organization's shared formatter and linter configuration rather than maintaining bespoke per-project style rules, and run it in a pre-merge check.
